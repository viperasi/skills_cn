# QA 会话

运行一次交互式 QA 会话。用户描述他们遇到的问题。你进行澄清，在后台探索代码库获取上下文，并提交持久的、以用户为中心的、使用项目领域语言的 GitHub Issue。

## 对于用户提出的每个问题

### 1. 倾听并适度澄清

让用户用自己的话描述问题。最多提出 **2-3 个简短的澄清问题**，重点关注：

- 他们期望什么 vs 实际发生了什么
- 复现步骤（如果不明显）
- 是否一致出现还是间歇性的

不要过度追问。如果描述已经足够清晰，可以直接提交 Issue。

### 2. 在后台探索代码库

在与用户交谈的同时，在后台启动一个 Agent（subagent_type=Explore）来了解相关领域。目标不是找到修复方案——而是：

- 学习该领域使用的领域语言（检查 UBIQUITOUS_LANGUAGE.md）
- 理解该功能应该做什么
- 识别面向用户的行为边界

这些上下文帮助你写出更好的 Issue——但 Issue 本身不应引用特定文件、行号或内部实现细节。

### 3. 评估范围：单个 Issue 还是拆分？

在提交之前，判断这是一个**单个 Issue** 还是需要**拆分成多个 Issue**。

在以下情况下拆分：

- 修复跨越多个独立领域（例如"表单验证错了 AND 成功消息缺失 AND 重定向坏了"）
- 有明确的、可以让不同人并行工作的独立关注点
- 用户描述的内容具有多个不同的失败模式或症状

在以下情况下保持为一个 Issue：

- 它是一个地方的一个错误行为
- 所有症状由同一个根本行为引起

### 4. 提交 GitHub Issue

使用 `gh issue create` 创建 Issue。不要要求用户先审查——直接提交并分享 URL。

Issue 必须是**持久的**——即使在重大重构后也应该有意义。从用户的角度来写。

#### 单个 Issue

使用以下模板：

```
## 发生了什么

[用简洁的语言描述用户经历的实际行为]

## 我期望的

[描述期望的行为]

## 复现步骤

1. [开发者可以遵循的具体、编号步骤]
2. [使用代码库的领域术语，而非内部模块名]
3. [包括相关的输入、标志或配置]

## 附加上下文

[来自用户或代码库探索的任何有助于框定问题的额外观察——例如"这仅在使用 Docker 层时发生，而非文件系统层"——使用领域语言但不引用文件]
```

#### 拆分（多个 Issue）

按依赖顺序创建 Issue（阻塞项先创建），这样你可以引用真实的 Issue 编号。

每个子 Issue 使用以下模板：

```
## 父 Issue

#<父 Issue 编号>（如果你创建了跟踪 Issue）或 "在 QA 会话中报告"

## 出了什么问题

[描述这个特定的行为问题——只是这一片，而非整个报告]

## 我期望的

[对这一特定片的期望行为]

## 复现步骤

1. [针对此 Issue 的特定步骤]

## 被阻塞于

- #<Issue编号>（如果此 Issue 在另一个 Issue 解决前无法处理）

或者 "无——可立即开始"（如果没有阻塞项）。

## 附加上下文

[与此片相关的任何额外观察]
```

在创建拆分时：

- **倾向于许多薄的 Issue 而非少数厚的**——每个 Issue 应该可以独立修复和验证
- **诚实地标记阻塞关系**——如果 Issue B 确实在 Issue A 修复之前无法测试，就如实说明。如果它们是独立的，将两个都标记为"无——可立即开始"
- **按依赖顺序创建 Issue**，这样你可以在"被阻塞于"中引用真实的 Issue 编号
- **最大化并行可能性**——目标是让多人（或智能体）可以同时认领不同的 Issue

#### 所有 Issue 正文的规则

- **不要包含文件路径或行号**——这些会过时
- **使用项目的领域语言**（检查 UBIQUITOUS_LANGUAGE.md 是否存在）
- **描述行为，而非代码**——"同步服务无法应用补丁"而非"applyPatch() 在第 42 行抛出异常"
- **复现步骤是必须的**——如果你无法确定，询问用户
- **保持简洁**——开发者应该能在 30 秒内读完 Issue

提交后，打印所有 Issue URL（附带阻塞关系摘要），并询问："下一个问题，还是我们结束了？"

### 5. 继续会话

持续进行直到用户表示完成。每个 Issue 是独立的——不要批量处理。
