---

当用户想要创建重构请求时，将调用此技能。你应该按以下步骤进行。如果你认为某些步骤不必要，可以跳过。

1. 让用户详细描述他们想要解决的问题，以及任何可能的解决方案。

2. 探索仓库以验证他们的断言并理解代码库的当前状态。

3. 询问他们是否考虑过其他方案，并向他们展示其他选项。

4. 就实现方案与用户进行面谈。要非常详细和全面。

5. 敲定实现的确切范围。确定你计划修改什么和计划不修改什么。

6. 查看代码库中此领域的测试覆盖情况。如果测试覆盖不足，询问用户的测试计划是什么。

7. 将实现拆分为一个小提交计划。记住 Martin Fowler 的建议："让每个重构步骤尽可能小，这样你始终能看到程序在运行。"

8. 使用重构计划创建一个 GitHub Issue。Issue 描述使用以下模板：

<refactor-plan-template>

## 问题陈述

开发者面临的问题，从开发者的角度描述。

## 解决方案

问题的解决方案，从开发者的角度描述。

## 提交

一个详细且长的实现计划。用简洁的英语写出计划，将实现拆分为尽可能小的提交。每个提交都应使代码库保持工作状态。

## 决策文档

已做出的实现决策列表。可以包括：

- 将要构建/修改的模块
- 将要修改的那些模块的接口
- 开发者的技术澄清
- 架构决策
- 模式变更
- API 约定
- 特定交互

不要包含特定文件路径或代码片段。它们可能很快就会过时。

## 测试决策

已做出的测试决策列表。包括：

- 什么是好测试的描述（只测试外部行为，而非实现细节）
- 哪些模块将被测试
- 测试的参考范例（即代码库中类似类型的测试）

## 不在范围内

此重构不在范围内的内容描述。

## 补充说明（可选）

关于重构的任何补充说明。

</refactor-plan-template>
