# 通用语言

从当前对话中提取并规范化领域术语，形成一致的词汇表，保存到本地文件。

## 流程

1. **扫描对话**，寻找与领域相关的名词、动词和概念
2. **识别问题**：
   - 同一个词用于不同的概念（歧义）
   - 不同词用于同一个概念（同义词）
   - 模糊或过载的术语
3. **提出规范词汇表**，带有明确的术语选择
4. **写入工作目录中的 `UBIQUITOUS_LANGUAGE.md`**，使用以下格式
5. **在对话中输出摘要**

## 输出格式

编写一个具有以下结构的 `UBIQUITOUS_LANGUAGE.md` 文件:

```md
# 通用语言

## 订单生命周期

| 术语          | 定义                               | 避免使用的别名      |
| ------------- | ---------------------------------- | ------------------- |
| **订单**       | 客户购买一个或多个商品的请求       | 购买、交易          |
| **发票**       | 交付后发送给客户的付款请求         | 账单、付款请求      |

## 人员

| 术语        | 定义                               | 避免使用的别名          |
| ----------- | ---------------------------------- | ----------------------- |
| **客户**     | 下订单的个人或组织                 | 客户端、买家、账户      |
| **用户**     | 系统中的认证身份                   | 登录、账户              |

## 关系

- 一个**发票**恰好属于一个**客户**
- 一个**订单**产生一个或多个**发票**

## 示例对话

> **开发者**："当**客户**下一个**订单**时，我们是否立即创建**发票**？"
> **领域专家**："不——**发票**仅在**履约单**确认后才生成。如果商品分多个**发货**发出，一个**订单**可以产生多个**发票**。"
> **开发者**："所以如果**发货**在发货前取消，它就没有**发票**？"
> **领域专家**："没错。**发票**的生命周期绑定在**履约单**上，而非**订单**。"

## 已标记的歧义

- "account" 被用于同时指代**客户**和**用户**——这是两个不同的概念：**客户**下订单，而**用户**是认证身份，可能代表也可能不代表**客户**。
```

## 规则

- **要有主见。** 当同一个概念存在多个词时，选择最好的一个，并将其他的列为避免使用的别名。
- **明确标记冲突。** 如果一个术语在对话中被模糊使用，在"已标记的歧义"部分中指出并给出清晰的建议。
- **仅包含与领域专家相关的术语。** 跳过模块或类的名称，除非它们在领域语言中有意义。
- **保持定义紧凑。** 最多一句话。定义它是什么，而非它做什么。
- **展示关系。** 使用粗体术语名称，并在明显时表达基数。
- **仅包含领域术语。** 跳过通用编程概念（array、function、endpoint），除非它们有领域特定的含义。
- **当出现自然簇时将术语分组到多个表格中**（例如按子领域、生命周期或参与者）。每组有自己的标题和表格。如果所有术语都属于一个统一的领域领域，一个表格就够了——不要强行分组。
- **写一个示例对话。** 开发者和领域专家之间的简短对话（3-5 轮交流），展示术语如何自然互动。对话应阐明相关概念之间的边界，并展示术语的精确使用方式。

## 示例对话

> **开发者**："如何在没有 Docker 的情况下测试**同步服务**？"

> **领域专家**："提供**文件系统层**代替**Docker 层**。它实现了相同的**沙箱服务**接口，但使用本地目录作为**沙箱**。"

> **开发者**："所以 **sync-in** 仍然创建一个**包**然后解压它？"

> **领域专家**："没错。**同步服务**不知道自己在跟哪个层对话。它调用 `exec` 和 `copyIn`——**文件系统层**只是把这些作为本地 shell 命令来运行。"

</example>

## 重新运行

当在同一对话中再次调用时：

1. 读取现有的 `UBIQUITOUS_LANGUAGE.md`
2. 纳入后续讨论中的任何新术语
3. 如果理解发生了变化，更新定义
4. 重新标记任何新歧义
5. 重写示例对话以纳入新术语
