# 设计接口

基于《软件设计的哲学》中的"设计两次"理念：你的第一个想法不太可能是最好的。生成多个截然不同的设计，然后进行比较。

## 工作流程

### 1. 收集需求

在设计之前，先了解：

- [ ] 这个模块解决什么问题？
- [ ] 调用者是谁？（其他模块、外部用户、测试）
- [ ] 关键操作是什么？
- [ ] 有什么约束？（性能、兼容性、已有模式）
- [ ] 哪些内容应该隐藏在内，哪些应该暴露？

询问："这个模块需要做什么？谁会使用它？"

### 2. 生成设计（并行子智能体）

使用 Task 工具同时启动 3 个以上的子智能体。每个必须产生一个**截然不同**的方法。

```
每个子智能体的提示模板：

为以下模块设计一个接口：[模块描述]

需求：[收集的需求]

此设计的约束：[为每个智能体分配不同的约束]
- 智能体 1："最小化方法数量——目标 1-3 个方法"
- 智能体 2："最大化灵活性——支持多种用例"
- 智能体 3："针对最常见的情况优化"
- 智能体 4："借鉴 [特定范式/库] 的风格"

输出格式：
1. 接口签名（类型/方法）
2. 使用示例（调用者如何使用）
3. 此设计隐藏了哪些内部实现
4. 此方法的权衡
```

### 3. 展示设计

对于每个设计，展示：

1. **接口签名** - 类型、方法、参数
2. **使用示例** - 调用者实际如何使用它
3. **隐藏的内容** - 保留在内部的复杂性

依次展示设计，以便用户在比较之前理解每种方法。

### 4. 比较设计

展示所有设计后，从以下方面进行比较：

- **接口简洁性**：更少的方法、更简单的参数
- **通用性 vs 专用性**：灵活性 vs 专注性
- **实现效率**：接口形态是否允许高效的内部实现？
- **深度**：小型接口隐藏大量复杂性（好）vs 大型接口附带浅薄实现（坏）
- **易于正确使用** vs **容易误用**

以散文形式讨论权衡，而非表格。突出设计差异最大的地方。

### 5. 综合

通常最好的设计会结合多种方案的洞察。询问：

- "哪个设计最适合你的主要用例？"
- "有没有其他设计中值得融入的元素？"

## 评估标准

来自《软件设计的哲学》：

**接口简洁性**：更少的方法、更简单的参数 = 更容易学习和正确使用。

**通用性**：能在不做改动的情况下处理未来的用例。但要注意不要过度泛化。

**实现效率**：接口形态是否允许高效的实现？还是会迫使内部实现变得别扭？

**深度**：小型接口隐藏大量复杂性 = 深模块（好）。大型接口附带浅薄实现 = 浅模块（避免）。

## 反模式

- 不要让子智能体生成相似的设计——强制要求截然不同
- 不要跳过比较——价值就在于对比
- 不要实现——这纯粹是关于接口形态
- 不要基于实现工作量来评估
