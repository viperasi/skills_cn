# 逻辑原型

一个微型交互式终端应用，让用户手动驱动状态模型。当问题是关于**业务逻辑、状态转换或数据形态**时使用——这类东西在纸面上看起来合理，但一旦通过真实案例推动就会感觉不对。

## 什么情况下适合此形态

- "我不确定这个状态机是否处理了 X 然后 Y 的边缘情况。"
- "这个数据模型是否能让我表示……的情况"
- "我想在写之前感受一下 API 应该是什么样子。"
- 任何用户想要**按按钮并观察状态变化**的情况。

如果问题是"这应该长什么样"——错误分支。使用 [UI.md](UI.md)。

## 流程

### 1. 说明问题

在写代码之前，写下你正在构建什么状态模型和什么问题的原型。一段话，在原型的 README 或文件顶部的注释中。一个回答了错误问题的逻辑原型是纯粹的浪费——使问题明确，以便稍后可以检查，无论用户是现在在看还是 AFK 后回来看。

### 2. 选择语言

使用主机项目使用的任何语言。如果项目没有明显的运行时（例如文档仓库），询问。

匹配项目现有的工具约定——不要为了原型添加新的包管理器或运行时。

### 3. 在可移植模块中隔离逻辑

将实际逻辑——回答问题的部分——放在一个小型的、纯粹的接口后面，可以稍后提取并放入真实代码库中。围绕它的 TUI 是临时的；逻辑模块不应该。

正确的形态取决于问题：

- **纯 reducer** — `(state, action) => state`。当操作是离散事件且状态是单值时合适。
- **状态机** — 显式状态和转换。当"哪些操作现在合法"是问题的一部分时合适。
- **对纯数据类型的一小组纯函数**。当没有隐式当前状态时合适——只有转换。
- **一个具有清晰方法面的类或模块**，当逻辑确实拥有持续的内部状态时。

选择最适合被问问题的形态，*而非*最容易连接到 TUI 的形态。保持纯净：没有 I/O，没有终端代码，没有用于控制流的 `console.log`。TUI 导入它并调用它；不反向流动。

这使原型在其自身生命周期之外有用。当问题得到回答后，经过验证的 reducer / 状态机 / 函数集可以被提升到真实模块中——TUI 外壳被删除。

### 4. 构建暴露状态的最小 TUI

将其构建为**轻量级 TUI**——在每个周期，清除屏幕（`console.clear()` / `print("\033[2J\033[H")` / 等价方式）并重新渲染整个画面。用户应该始终看到一个稳定的视图，而不是不断增长的滚动历史。

每帧有两个部分，按此顺序：

1. **当前状态**，美化打印且对差异友好（每行一个字段，或格式化的 JSON）。对字段名或章节标题使用**粗体**，对不太重要的上下文（时间戳、ID、派生值）使用**暗色**。原生 ANSI 转义码即可——`\x1b[1m` 粗体、`\x1b[2m` 暗色、`\x1b[0m` 重置。除非项目中已有样式库，否则无需引入。
2. **键盘快捷键**，在底部列出：`[a] 添加用户  [d] 删除用户  [t] 推进时间  [q] 退出`。粗体键名，暗色描述，或反过来——怎么看起来清晰就怎么来。

行为：

1. **初始化状态**——单个内存中对象/结构体。启动时渲染第一帧。
2. **一次读取一个按键（或一行）**，分发到修改状态的处理程序。
3. **每次操作后重新渲染**完整帧——不要追加，替换。
4. **循环直到退出。**

整个帧应该能放在一个屏幕上。

### 5. 使其可通过一条命令运行

将脚本添加到项目现有任务运行器中（`package.json` scripts、`Makefile`、`justfile`、`pyproject.toml`）。用户应该运行 `pnpm run <prototype-name>` 或等价方式——绝不需要记住路径。

如果主机项目没有任务运行器，只需将命令放在原型的 README 顶部。

### 6. 交付

给用户运行命令。他们会自己驱动它；有趣的时刻是当他们说"等等，那不应该可能"或"嗯，我以为 X 会不同"时——这些是*想法*中的 bug，这正是重点。如果他们想要添加新操作，添加它们。原型会演变。

### 7. 捕获答案

当原型完成其工作后，问题的答案就是唯一值得保留的东西。如果用户在，询问它教会了他们什么。如果不在，在原型旁边留下一个 `NOTES.md`，以便在原型被删除之前可以填写（或由你在观看了会话后填写）答案。

## 反模式

- **不要添加测试。** 需要测试的原型不再是原型。
- **不要连接到真实数据库。** 使用内存中存储，除非问题专门关于持久化。
- **不要通用化。** 没有"如果以后我们想支持 X 怎么办"。原型回答一个问题。
- **不要把逻辑和 TUI 混在一起。** 如果 reducer / 状态机引用了 `console.log`、提示或终端转义码，它就不再可移植。将 TUI 保持为纯模块上的薄外壳。
- **不要将 TUI 外壳投入生产。** 外壳是为从终端手动驱动优化的。其后的逻辑模块是值得保留的部分。
