# ADR 格式

ADR 存放在 `docs/adr/` 中，使用顺序编号：`0001-slug.md`、`0002-slug.md` 等。

按需创建 `docs/adr/` 目录——仅在需要第一个 ADR 时创建。

## 模板

```md
# {决策的简短标题}

{1-3 句话：背景是什么，我们决定了什么，为什么。}
```

仅此而已。一个 ADR 可以只有一段话。价值在于记录 *做出了决策* 和 *为什么*——不在于填满各个部分。

## 可选章节

仅当能带来真正价值时才包括这些内容。大多数 ADR 不需要它们。

- **状态** 前置元数据（`proposed | accepted | deprecated | superseded by ADR-NNNN`）——当决策被重新审视时有用
- **考虑的选项** —— 仅当被拒绝的替代方案值得记住时
- **后果** —— 仅当非显而易见的下游影响需要被指出时

## 编号

扫描 `docs/adr/` 中现有的最高编号并加一。

## 何时提供 ADR

以下三个条件必须全部满足：

1. **难以撤销** — 以后改变主意的成本是有意义的
2. **没有上下文会令人惊讶** — 未来的读者看到代码后会想"他们到底为什么这样做？"
3. **真正的权衡结果** — 确实有真正的替代方案，你出于特定原因选择了其中一个

如果一个决策容易撤销，跳过——你直接撤销就行了。如果它不会令人惊讶，没有人会想知道为什么。如果根本没有真正的替代方案，那也没有什么可以记录的，除了"我们做了显而易见的事情"。

### 什么符合资格

- **架构形态。** "我们使用 monorepo。" "写模型是事件溯源的，读模型投射到 Postgres。"
- **上下文之间的集成模式。** "订购和计费通过领域事件通信，而非同步 HTTP。"
- **具有锁定效应的技术选择。** 数据库、消息总线、认证提供者、部署目标。不是每个库——只是那些需要花一个季度才能替换掉的。
- **边界和范围决策。** "客户数据属于客户上下文所有；其他上下文只通过 ID 引用。" 明确的"不做"和"做"一样有价值。
- **对明显路径的有意偏离。** "我们使用手动 SQL 而不是 ORM，因为 X。" 任何合理读者会假设相反的事情。这些能阻止下一个工程师"修复"某个故意的设计。
- **代码中不可见的约束。** "由于合规要求，我们不能使用 AWS。" "由于合作伙伴 API 合同，响应时间必须低于 200ms。"
- **当拒绝非显而易见的替代方案时。** 如果你考虑了 GraphQL 但出于微妙原因选择了 REST，记录下来——否则六个月后有人会再次提议 GraphQL。
