# CONTEXT.md 格式

## 结构

```md
# {上下文名称}

{一两句话描述这个上下文是什么以及它为什么存在。}

## 语言

**订单**:
{该术语的简洁描述}
_避免使用_: 购买、交易

**发票**:
交付后发送给客户的付款请求。
_避免使用_: 账单、付款请求

**客户**:
下订单的个人或组织。
_避免使用_: 客户端、买家、账户

## 关系

- 一个**订单**产生一个或多个**发票**
- 一个**发票**恰好属于一个**客户**

## 示例对话

> **开发者**："当**客户**下一个**订单**时，我们是否立即创建**发票**？"
> **领域专家**："不——**发票**仅在**履约单**确认后才生成。"

## 已标记的歧义

- "account" 被用于同时指代**客户**和**用户**——已解决：这是两个不同的概念。
```

## 规则

- **要有主见。** 当同一个概念存在多个词时，选择最好的一个，将其他的列为避免使用的别名。
- **明确标记冲突。** 如果一个术语被模糊使用，在"已标记的歧义"中明确指出并给出清晰的解决方案。
- **保持定义紧凑。** 最多一句话。定义它是什么，而非它做什么。
- **展示关系。** 使用粗体术语名称，并在明显时表达基数。
- **仅包含特定于此项目上下文的术语。** 通用编程概念（超时、错误类型、工具模式）不属于，即使项目广泛使用它们。在添加一个术语之前，问：这是此上下文独有的概念，还是一个通用编程概念？只有前者才属于。
- **当出现自然簇时将术语分组到子标题下。** 如果所有术语都属于一个统一的领域，平坦列表也可以。
- **写一个示例对话。** 开发者和领域专家之间的对话，展示术语如何自然互动，并澄清相关概念之间的边界。

## 单上下文 vs 多上下文仓库

**单上下文（大多数仓库）：** 一个 `CONTEXT.md` 在仓库根目录。

**多上下文：** 仓库根目录有一个 `CONTEXT-MAP.md`，列出各个上下文、它们的位置以及它们之间的关系：

```md
# 上下文映射

## 上下文

- [订购](./src/ordering/CONTEXT.md) — 接收和跟踪客户订单
- [计费](./src/billing/CONTEXT.md) — 生成发票和处理付款
- [履约](./src/fulfillment/CONTEXT.md) — 管理仓库拣货和发货

## 关系

- **订购 → 履约**: 订购发出 `OrderPlaced` 事件；履约消费它们以开始拣货
- **履约 → 计费**: 履约发出 `ShipmentDispatched` 事件；计费消费它们以生成发票
- **订购 ↔ 计费**: 共享 `CustomerId` 和 `Money` 类型
```

技能推断适用哪种结构：

- 如果存在 `CONTEXT-MAP.md`，读取它以找到各个上下文
- 如果只存在根 `CONTEXT.md`，则为单上下文
- 如果两者都不存在，在第一个术语解决时按需创建根 `CONTEXT.md`

当存在多个上下文时，推断当前主题与哪一个相关。如果不清楚，询问。
