# 范围外知识库

仓库中的 `.out-of-scope/` 目录存储被拒绝功能请求的持久记录。它有两个目的：

1. **机构记忆** — 为什么一个功能被拒绝，这样当 Issue 关闭时推理不会丢失
2. **去重** — 当新 Issue 进来且匹配先前的拒绝时，技能可以展示先前的决策而不是重新讨论它

## 目录结构

```
.out-of-scope/
├── dark-mode.md
├── plugin-system.md
└── graphql-api.md
```

每个**概念**一个文件，而非每个 Issue 一个文件。请求相同事物的多个 Issue 分组到一个文件下。

## 文件格式

文件应以轻松、可读的风格编写——更像一个简短的设计文档而非数据库条目。使用段落、代码示例和例子使推理清晰并对第一次遇到它的人有用。

```markdown
# 深色模式

此项目不支持深色模式或面向用户的主题。

## 为什么不在范围内

渲染管道假设 `ThemeConfig` 中定义了一个单一调色板。
支持多个主题将需要：

- 一个包裹整个组件树的主题上下文提供者
- 每个组件的主题感知样式解析
- 用户主题偏好的持久层

这是一个重大的架构变更，与项目关注内容创作的重点不一致。
主题是嵌入或重新分发输出的下游消费者的关注点。

```ts
// 当前的 ThemeConfig 接口不是为运行时切换设计的：
interface ThemeConfig {
  colors: ColorPalette; // 单一调色板，在构建时解析
  fonts: FontStack;
}
```

## 先前请求

- #42 — "添加深色模式支持"
- #87 — "夜间主题以提升可访问性"
- #134 — "深色主题选项"
```

### 命名文件

对概念使用简短、描述性的 kebab-case 名称：`dark-mode.md`、`plugin-system.md`、`graphql-api.md`。名称应足够可识别，使浏览目录的人不必打开文件就能理解什么被拒绝了。

### 编写原因

原因应是实质性的——不是"我们不想要这个"而是为什么。好的原因引用：

- 项目范围或理念（"此项目重点在 X；主题是下游关注点"）
- 技术约束（"支持这个将需要 Y，它与我们的 Z 架构冲突"）
- 战略决策（"我们选择使用 A 而不是 B 因为……"）

原因应是持久的。避免引用临时情况（"我们现在太忙了"）——那些不是真正的拒绝，而是推迟。

## 何时检查 `.out-of-scope/`

分类期间（步骤 1：收集上下文），读取 `.out-of-scope/` 中的所有文件。评估新 Issue 时：

- 检查请求是否匹配现有的范围外概念
- 匹配是按概念相似性，而非关键词——"night theme" 匹配 `dark-mode.md`
- 如果有匹配，向维护者展示："这类似于 `.out-of-scope/dark-mode.md`——我们之前因为 [原因] 拒绝了。你仍然觉得一样吗？"

维护者可能：

- **确认** — 新 Issue 添加到现有文件的"先前请求"列表，然后关闭
- **重新考虑** — 范围外文件被删除或更新，Issue 进入正常分类流程
- **不同意** — Issue 相关但不同，继续正常分类

## 何时写入 `.out-of-scope/`

仅当一个 **enhancement**（而非 bug）被拒绝为 `wontfix` 时。流程：

1. 维护者决定一个功能请求不在范围内
2. 检查匹配的 `.out-of-scope/` 文件是否已存在
3. 如果是：将新 Issue 追加到"先前请求"列表
4. 如果否：创建一个新文件，包含概念名称、决策、原因和第一个先前请求
5. 在 Issue 上发布一条评论解释决策并提及 `.out-of-scope/` 文件
6. 以 `wontfix` 标签关闭 Issue

## 更新或移除范围外文件

如果维护者改变了对先前被拒绝概念的想法：

- 删除 `.out-of-scope/` 文件
- 技能不需要重新打开旧的 Issue——它们是历史记录
- 触发重新考虑的新 Issue 进入正常分类流程
