# 分类

通过一个小型的分类角色状态机在项目问题跟踪器上移动 Issue。

分类过程中发布到问题跟踪器的每条评论或 Issue **必须**以此免责声明开头：

```
> *这是由 AI 在分类过程中生成的。*
```

## 参考文档

- [AGENT-BRIEF.md](AGENT-BRIEF.md) — 如何编写持久的智能体简报
- [OUT-OF-SCOPE.md](OUT-OF-SCOPE.md) — `.out-of-scope/` 知识库如何工作

## 角色

两个**类别**角色：

- `bug` — 某事物损坏了
- `enhancement` — 新功能或改进

五个**状态**角色：

- `needs-triage` — 维护者需要评估
- `needs-info` — 等待报告者提供更多信息
- `ready-for-agent` — 完全指定，可供 AFK 智能体使用
- `ready-for-human` — 需要人类实现
- `wontfix` — 不会被处理

每个已分类的 Issue 应携带恰好一个类别角色和一个状态角色。如果状态角色冲突，标记并询问维护者，然后再做其他任何事。

这些是规范角色名称——问题跟踪器中使用的实际标签字符串可能不同。映射应该已经提供给你——如果没有，请运行 `/setup-matt-pocock-skills`。

状态转换：一个无标签的 Issue 通常首先进入 `needs-triage`；从那里移动到 `needs-info`、`ready-for-agent`、`ready-for-human` 或 `wontfix`。一旦报告者回复，`needs-info` 返回到 `needs-triage`。维护者可以随时覆盖——标记看起来不寻常的转换并在继续之前询问。

## 调用

维护者调用 `/triage` 并用自然语言描述他们想要什么。解读请求并行动。示例：

- "显示需要我关注的任何内容"
- "让我们看看 #42"
- "将 #42 移至 ready-for-agent"
- "哪些已准备好给智能体认领？"

## 显示需要关注的内容

查询问题跟踪器并展示三个桶，最旧的优先：

1. **无标签** — 从未分类。
2. **`needs-triage`** — 评估进行中。
3. **`needs-info` 且自上次分类记录以来有报告者活动** — 需要重新评估。

显示计数和每个 Issue 的一行摘要。让维护者选择。

## 分类一个特定 Issue

1. **收集上下文。** 读取完整的 Issue（正文、评论、标签、报告者、日期）。解析任何之前的分类记录，这样你不会重新询问已解决的问题。使用项目的领域词汇表探索代码库，尊重你涉及领域的 ADR。读取 `.out-of-scope/*.md` 并展示任何与此 Issue 相似的先前拒绝。

2. **推荐。** 告诉维护者你的类别和状态推荐及推理，加上与 Issue 相关的简要代码库摘要。等待指示。

3. **复现（仅限 bug）。** 在任何盘问之前，尝试复现：阅读报告者的步骤，追踪相关代码，运行测试或命令。报告发生了什么——带有代码路径的成功复现、失败的复现或细节不足（一个强烈的 `needs-info` 信号）。确认的复现会做出更强得多的智能体简报。

4. **盘问（如需要）。** 如果 Issue 需要充实，运行一次 `/grill-with-docs` 会话。

5. **应用结果：**
   - `ready-for-agent` — 发布智能体简报评论（[AGENT-BRIEF.md](AGENT-BRIEF.md)）。
   - `ready-for-human` — 与智能体简报结构相同，但注明为什么不能委派（判断调用、外部访问、设计决策、手动测试）。
   - `needs-info` — 发布分类记录（下方模板）。
   - `wontfix`（bug）— 礼貌解释，然后关闭。
   - `wontfix`（enhancement）— 写入 `.out-of-scope/`，从评论链接到它，然后关闭（[OUT-OF-SCOPE.md](OUT-OF-SCOPE.md)）。
   - `needs-triage` — 应用角色。如果有部分进展，可选评论。

## 快速状态覆盖

如果维护者说"将 #42 移至 ready-for-agent"，信任他们并直接应用角色。确认你将要做什么（角色更改、评论、关闭），然后行动。跳过盘问。如果移至 `ready-for-agent` 且没有盘问会话，询问他们是否想编写智能体简报。

## 需要信息的模板

```markdown
## 分类记录

**目前为止我们已确定的：**

- 点 1
- 点 2

**我们仍需你提供（@report）：**

- 问题 1
- 问题 2
```

捕获盘问期间解决的所有内容在"已确定的"下，这样工作不会丢失。问题必须具体和可操作，而非"请提供更多信息"。

## 恢复之前的会话

如果 Issue 上存在之前的分类记录，读取它们，检查报告者是否已回答任何未解决的问题，并在继续之前展示更新后的情况。不要重新询问已解决的问题。
