# 诊断

应对疑难 bug 的规范方法。只有在有明确理由时才跳过阶段。

在探索代码库时，使用项目的领域词汇表来获得相关模块的清晰心智模型，并检查你涉及领域的 ADR。

## 阶段 1 —— 构建反馈循环

**这就是核心技能。** 其他一切都是机械的。如果你有一个快速、确定性的、智能体可运行的关于此 bug 的通过/失败信号，你就会找到原因——二分法、假设测试和插桩都只是在消费这个信号。如果你没有，盯着代码看多久都没用。

在此阶段投入不成比例的努力。**要主动，要有创意，拒绝放弃。**

### 构建的方法——大致按以下顺序尝试

1. **失败的测试**，在与 bug 接触的任何接缝处——单元测试、集成测试、端到端测试。
2. **Curl / HTTP 脚本**，针对运行中的开发服务器。
3. **CLI 调用**，带有固定输入，与已知正确的快照进行输出对比。
4. **无头浏览器脚本**（Playwright / Puppeteer）——驱动 UI，断言 DOM/控制台/网络。
5. **回放捕获的追踪。** 保存真实的网络请求 / 负载 / 事件日志到磁盘；在隔离中通过该代码路径回放。
6. **一次性测试工具。** 启动一个最小的系统子集（一个服务、模拟的依赖），通过单个函数调用来执行 bug 代码路径。
7. **属性 / 模糊循环。** 如果 bug 是"有时输出错误"，运行 1000 个随机输入并寻找失败模式。
8. **二分测试工具。** 如果 bug 出现在两个已知状态之间（提交、数据集、版本），自动化"在状态 X 启动，检查，重复"，这样你就可以 `git bisect run`。
9. **差分循环。** 在旧版本 vs 新版本（或两种配置）上运行相同输入并比较输出。
10. **HITL bash 脚本。** 最后的手段。如果必须由人点击，用 `scripts/hitl-loop.template.sh` 来驱动他们，循环仍然是结构化的。捕获的输出会反馈给你。

构建正确的反馈循环，bug 就已经修好 90% 了。

### 优化循环本身

将循环视为产品。一旦你有了一个循环，问：

- 可以更快吗？（缓存设置、跳过无关初始化、缩小测试范围）
- 信号可以更清晰吗？（断言特定症状，而非"没崩溃"）
- 可以更确定性吗？（固定时间、种子 RNG、隔离文件系统、冻结网络）

一个 30 秒的脆弱循环仅比没有循环好一点。一个 2 秒的确定性循环是调试的超能力。

### 非确定性 bug

目标不是干净的复现，而是**更高的复现率**。循环触发 100 次，并行化，增加压力，缩小时间窗口，注入延迟。50% 的不稳定性是可以调试的；1% 不行——持续提高比率直到可以调试。

### 当确实无法构建循环时

停下来并明确说明。列出你尝试了什么。向用户请求：(a) 访问能复现它的任何环境，(b) 一个捕获的产物（HAR 文件、日志转储、核心转储、带时间戳的屏幕录制），或 (c) 添加临时生产插桩的权限。没有循环，**不要**继续进入假设阶段。

在拥有让你信服的循环之前，不要进入阶段 2。

## 阶段 2 —— 复现

运行循环。观察 bug 出现。

确认：

- [ ] 循环产生了**用户**描述的失败模式——而不是碰巧在附近的另一个失败。错误的 bug = 错误的修复。
- [ ] 失败在多次运行中是可复现的（或对于非确定性 bug，以足够高的比率复现以进行调试）。
- [ ] 你已经捕获了确切的症状（错误消息、错误的输出、慢的时间），以便后续阶段可以验证修复确实解决了它。

在复现 bug 之前不要继续。

## 阶段 3 —— 假设

在测试任何假设之前生成 **3-5 个按优先级排列的假设**。单个假设生成会被第一个看似合理的想法锚定。

每个假设必须是**可证伪的**：说明它做出的预测。

> 格式："如果 <X> 是原因，那么 <改变 Y> 将使 bug 消失 / <改变 Z> 将使其更严重。"

如果你无法说明预测，假设只是一种感觉——丢弃或精炼。

**在测试之前将排序列表展示给用户。** 他们通常有领域知识可以立即重新排序（"我们刚刚部署了一个对 3 号的更改"），或者知道他们已排除的假设。低成本检查点，省大量的时间。但不要因此阻塞——如果用户 AFK，就按你的排序继续。

## 阶段 4 —— 插桩

每次探测必须映射到阶段 3 中的一个特定预测。**一次只改变一个变量。**

工具偏好：

1. **调试器 / REPL 检查**，如果环境支持。一个断点胜过十个日志。
2. **定向日志**，放在能区分假设的边界处。
3. 永远不要"记录所有日志然后搜索"。

**用唯前缀标记每个调试日志**，例如 `[DEBUG-a4f2]`。结束时的清理变成一次简单的 grep。无标记的日志保留；有标记的日志删除。

**性能分支。** 对于性能回归，日志通常是错的方法。相反：建立一个基线测量（时间测试工具、`performance.now()`、分析器、查询计划），然后二分。先测量，再修复。

## 阶段 5 —— 修复 + 回归测试

**在修复之前**写回归测试——但仅当存在**正确的接缝**时。

正确的接缝是指测试在调用点真实地执行**真实 bug 模式**的地方。如果唯一可用的接缝太浅（当 bug 需要多个调用者时的单调用者测试，或无法复制触发 bug 链路的单元测试），那里的回归测试会给人虚假的信心。

**如果不存在正确接缝，这本身就是发现。** 记录下来。代码库架构使此 bug 无法被锁定。标记此事，以备下个阶段。

如果存在正确的接缝：

1. 将最小化的复现转换为该接缝处的失败测试。
2. 观察它失败。
3. 应用修复。
4. 观察它通过。
5. 对原始（未最小化的）场景重新运行阶段 1 的反馈循环。

## 阶段 6 —— 清理 + 事后分析

完成前必须：

- [ ] 原始复现不再复现（重新运行阶段 1 循环）
- [ ] 回归测试通过（或接缝缺失已被记录）
- [ ] 所有 `[DEBUG-...]` 插桩已移除（grep 前缀）
- [ ] 一次性原型已删除（或移至明确标记的调试位置）
- [ ] 被证正确的假设在提交 / PR 消息中说明——这样下一个调试者可以学习

**然后问：什么能防止这个 bug？** 如果答案涉及架构变更（没有好的测试接缝、羁绊的调用者、隐藏的耦合），将具体情况移交给 `/improve-codebase-architecture` 技能。在修复完成后提出建议，而非之前——现在你比开始时掌握了更多信息。
