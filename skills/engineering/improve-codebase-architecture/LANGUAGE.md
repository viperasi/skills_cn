# 语言

此技能提出的每条建议使用的共享词汇。精确使用这些术语——不要用"component"、"service"、"API"或"boundary"替换。一致的语言就是全部意义。

## 术语

**模块**
任何有接口和实现的东西。故意不特定于规模——同等地适用于函数、类、包或跨层级切片。
_避免使用_: unit、component、service。

**接口**
调用者为正确使用模块必须知道的一切。包括类型签名，但也包括不变量、顺序约束、错误模式、所需配置和性能特征。
_避免使用_: API、签名（太窄——这些只指类型级表面）。

**实现**
模块内部的内容——它的代码体。与**适配器**不同：一个东西可以是一个具有大实现的小适配器（Postgres 仓库）或一个具有小实现的大适配器（内存假对象）。当话题是接缝时选用"适配器"；否则用"实现"。

**深度**
接口处的杠杆——调用者（或测试）每单位需要学习的接口可行使的行为量。当一个模块有大量行为隐藏在一个小接口后面时，它是**深的**。当接口几乎和实现一样复杂时，模块是**浅的**。

**接缝** _(源自 Michael Feathers)_
一个你可以在不原地编辑的情况下改变行为的地方。模块接口所在的*位置*。选择把接缝放在哪里是一个独立的设计决策，不同于接缝后面放什么。
_避免使用_: boundary（与 DDD 的限界上下文过载）。

**适配器**
在接缝处满足接口的具体东西。描述的是*角色*（它填充了哪个槽），而非实质（里面有什么）。

**杠杆**
调用者从深度中获得的收益。每单位需要学习的接口获得更多的能力。一个实现在 N 个调用点和 M 个测试中得到回报。

**局部性**
维护者从深度中获得的收益。变更、bug、知识和验证集中在一个地方而非分散在调用者之间。一次修复，到处修复。

## 原则

- **深度是接口的属性，而非实现的属性。** 深模块可以内部由小的、可模拟的、可替换的部件组成——它们只是不是接口的一部分。模块可以有**内部接缝**（其实现私有，由其自身测试使用）以及其接口处的**外部接缝**。
- **删除测试。** 想象删除该模块。如果复杂性消失，该模块没有隐藏任何东西（它是个传递通道）。如果复杂性在 N 个调用者中重新出现，该模块在发挥作用。
- **接口就是测试面。** 调用者和测试穿过同一个接缝。如果你想测试*超出*接口的内容，模块的形态可能不对。
- **一个适配器意味着假设的接缝。两个适配器意味着真正的接缝。** 除非有什么东西确实在接缝处变化，否则不要引入接缝。

## 关系

- 一个**模块**恰好有一个**接口**（它呈现给调用者和测试的表面）。
- **深度**是**模块**的一个属性，相对于其**接口**来衡量。
- **接缝**是**模块**的**接口**所在之处。
- 一个**适配器**位于**接缝**处并满足**接口**。
- **深度**为调用者产生**杠杆**，为维护者产生**局部性**。

## 被拒绝的框架

- **深度作为实现行数与接口行数之比**（Ousterhout）：奖励填充实现。我们改用深度即杠杆。
- **"接口"作为 TypeScript `interface` 关键字或类的公共方法**：太窄——这里的接口包括调用者必须知道的每一件事。
- **"边界"**：与 DDD 的限界上下文过载。说**接缝**或**接口**。
