# 改进代码库架构

发现架构摩擦并提出**深化机会**——将浅模块变为深模块的重构。目标是可测试性和 AI 可导航性。

## 词汇表

在每条建议中精确使用这些术语。一致的语言就是关键——不要偏离到 "component"、"service"、"API" 或 "boundary"。完整定义见 [LANGUAGE.md](LANGUAGE.md)。

- **模块** — 任何有接口和实现的东西（函数、类、包、切片）。
- **接口** — 调用者使用模块必须知道的一切：类型、不变量、错误模式、顺序、配置。不仅仅是类型签名。
- **实现** — 内部的代码。
- **深度** — 接口处的杠杆：大量行为隐藏在小接口后面。**深** = 高杠杆。**浅** = 接口几乎和实现一样复杂。
- **接缝** — 接口所在的位置；一个可以在不原地编辑的情况下改变行为的地方。（使用此术语，而非 "boundary"。）
- **适配器** — 在接缝处满足接口的具体东西。
- **杠杆** — 调用者从深度中获得的收益。
- **局部性** — 维护者从深度中获得的收益：变更、bug、知识集中在一个地方。

核心原则（完整列表见 [LANGUAGE.md](LANGUAGE.md)）：

- **删除测试**：想象删除该模块。如果复杂性消失，它是个传递通道。如果复杂性在 N 个调用者中重新出现，它在发挥作用。
- **接口就是测试面。**
- **一个适配器 = 假设的接缝。两个适配器 = 真正的接缝。**

此技能以项目的领域模型为*指导*。领域语言为好接缝提供名称；ADR 记录了技能不应重新讨论的决策。

## 流程

### 1. 探索

首先阅读项目的领域词汇表和与你所涉及领域相关的任何 ADR。

然后使用 Agent 工具，`subagent_type=Explore` 遍历代码库。不要依循僵化的启发式——有机地探索并记录你遇到摩擦的地方：

- 哪里理解一个概念需要在许多小模块之间来回跳转？
- 哪些模块是**浅的**——接口几乎和实现一样复杂？
- 哪里纯函数仅仅是为了可测试性而被提取出来，但真正的 bug 隐藏在它们如何被调用（没有**局部性**）？
- 哪里紧密耦合的模块在其接缝之间泄漏？
- 代码库中哪些部分未经测试，或难以通过其当前接口进行测试？

对任何你怀疑是浅模块的东西应用**删除测试**：删除它会集中复杂性，还是仅仅移动它？"是的，集中"是你想要的信号。

### 2. 呈现候选

呈现编号的深化机会列表。对于每个候选：

- **文件** — 涉及哪些文件/模块
- **问题** — 为什么当前架构造成摩擦
- **解决方案** — 用简洁的英语描述将要改变什么
- **好处** — 用局部性和杠杆来解释，以及测试会如何改善

**对领域使用 CONTEXT.md 词汇表，对架构使用 [LANGUAGE.md](LANGUAGE.md) 词汇表。** 如果 `CONTEXT.md` 定义了 "Order"，谈论"订单接收模块"——而非 "FooBarHandler"，也非 "Order service"。

**ADR 冲突**：如果一个候选与现有 ADR 相矛盾，仅当摩擦足够真实以至于有理由重新审视 ADR 时才提出来。清晰地标记（例如 *"与 ADR-0007 矛盾——但值得重新审视，因为……"*）。不要列出每个 ADR 禁止的理论重构。

不要提出接口。问用户："你想探索哪一个？"

### 3. 盘问循环

一旦用户选择了一个候选，进入盘问对话。与他们一起走过设计树——约束、依赖、深化模块的形态、接缝后面有什么、哪些测试会保留下来。

副作用会在决策固化时即时发生：

- **为不在 `CONTEXT.md` 中的概念命名的深化模块？** 将术语添加到 `CONTEXT.md`——与 `/grill-with-docs` 相同的规范（参见 [CONTEXT-FORMAT.md](../grill-with-docs/CONTEXT-FORMAT.md)）。如果不存在则按需创建文件。
- **在对话中精炼模糊术语？** 即时更新 `CONTEXT.md`。
- **用户以具有承载力的理由拒绝候选？** 提供一个 ADR，框架为：*"要我将其记录为 ADR，这样未来的架构审查就不会重新建议它吗？"* 仅当该理由确实会被未来的探索者需要以避免重新建议相同的内容时才提供——跳过短暂的理由（"现在不值得"）和显而易见的理由。参见 [ADR-FORMAT.md](../grill-with-docs/ADR-FORMAT.md)。
- **想要为深化模块探索替代接口？** 参见 [INTERFACE-DESIGN.md](INTERFACE-DESIGN.md)。
