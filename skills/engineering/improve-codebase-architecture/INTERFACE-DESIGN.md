# 接口设计

当用户想要为选定的深化候选探索替代接口时，使用此并行子智能体模式。基于"设计两次"（Ousterhout）——你的第一个想法不太可能是最好的。

使用 [LANGUAGE.md](LANGUAGE.md) 中的词汇——**module**、**interface**、**seam**、**adapter**、**leverage**。

## 流程

### 1. 界定问题空间

在启动子智能体之前，为选定的候选写一个面向用户的问题空间说明：

- 任何新接口需要满足的约束
- 它将依赖的依赖项，以及它们属于哪个类别（参见 [DEEPENING.md](DEEPENING.md)）
- 一个大致的说明性代码草图来使约束具体化——不是提案，只是使约束变得具体的方式

将此展示给用户，然后立即进入步骤 2。用户阅读和思考的同时，子智能体并行工作。

### 2. 启动子智能体

使用 Agent 工具并行启动 3 个以上子智能体。每个必须为深化模块产生一个**截然不同**的接口。

用单独的技术简报提示每个子智能体（文件路径、耦合细节、来自 [DEEPENING.md](DEEPENING.md) 的依赖类别、接缝背后的内容）。简报独立于步骤 1 中面向用户的问题空间说明。给每个智能体一个不同的设计约束：

- 智能体 1："最小化接口——目标 1-3 个入口点。最大化每个入口点的杠杆。"
- 智能体 2："最大化灵活性——支持许多用例和扩展。"
- 智能体 3："针对最常见的调用者优化——使默认情况变得微不足道。"
- 智能体 4（如适用）："围绕跨接缝依赖项的端口和适配器设计。"

在简报中包含 [LANGUAGE.md](LANGUAGE.md) 词汇和 CONTEXT.md 词汇，使每个子智能体用架构语言和项目的领域语言一致地命名。

每个子智能体输出：

1. 接口（类型、方法、参数——加上不变量、顺序、错误模式）
2. 展示调用者如何使用它的使用示例
3. 实现在接缝背后隐藏了什么
4. 依赖策略和适配器（参见 [DEEPENING.md](DEEPENING.md)）
5. 权衡——哪里杠杆高，哪里薄弱

### 3. 展示和比较

按顺序展示设计，以便用户吸收每一个，然后用散文进行比较。按**深度**（接口处的杠杆）、**局部性**（变更集中的地方）和**接缝位置**进行对比。

比较之后，给出你自己的推荐：你认为哪个设计最强，为什么。如果不同设计的元素会结合得很好，提出一个混合方案。要有主见——用户想要的是强烈的解读，而不是菜单。
