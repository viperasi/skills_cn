# 编写技能

## 流程

1. **收集需求** - 询问用户：
   - 技能涵盖什么任务/领域？
   - 应处理哪些特定用例？
   - 需要可执行脚本还是只需指令？
   - 任何需要包含的参考材料？

2. **起草技能** - 创建：
   - 具有简洁指令的 SKILL.md
   - 如果内容超过 500 行，附加参考文件
   - 如果需要确定性操作，编写工具脚本

3. **与用户审阅** - 展示草稿并询问：
   - 这覆盖了你的用例吗？
   - 有什么遗漏或不清楚的吗？
   - 有哪个部分应该更详细或更简洁吗？

## 技能结构

```
skill-name/
├── SKILL.md           # 主要指令（必需）
├── REFERENCE.md       # 详细文档（如需要）
├── EXAMPLES.md        # 使用示例（如需要）
└── scripts/           # 工具脚本（如需要）
    └── helper.js
```

## SKILL.md 模板

```md
---
name: skill-name
description: 能力简述。当[特定触发条件]时使用。
---

# 技能名称

## 快速开始

[最小工作示例]

## 工作流

[复杂任务的分步流程，带检查清单]

## 高级功能

[链接到单独文件：参见 [REFERENCE.md](REFERENCE.md)]
```

## 描述要求

描述是**你的智能体在选择加载哪个技能时唯一看到的东西**。它与所有其他已安装技能一起出现在系统提示中。你的智能体阅读这些描述并根据用户请求选择相关技能。

**目标**：给智能体足够的信息来了解：

1. 此技能提供什么能力
2. 何时/为何触发它（特定关键词、上下文、文件类型）

**格式**：

- 最多 1024 字符
- 用第三人称写
- 第一句：它做什么
- 第二句："当[特定触发条件]时使用"

**好的示例**：

```
从 PDF 文件中提取文本和表格、填写表单、合并文档。当处理 PDF 文件或用户提到 PDF、表单或文档提取时使用。
```

**坏的示例**：

```
帮助处理文档。
```

坏示例让智能体无法区分此技能与其他文档技能。

## 何时添加脚本

在以下情况下添加工具脚本：

- 操作是确定性的（验证、格式化）
- 相同的代码会重复生成
- 错误需要显式处理

脚本节省 token 并比生成代码提高可靠性。

## 何时拆分文件

在以下情况下拆分为单独文件：

- SKILL.md 超过 100 行
- 内容有不同的领域（财务 vs 销售 schema）
- 高级功能很少需要

## 审阅检查清单

起草后，验证：

- [ ] 描述包含触发条件（"当……时使用"）
- [ ] SKILL.md 在 100 行以内
- [ ] 没有时间敏感信息
- [ ] 术语一致
- [ ] 包含具体示例
- [ ] 引用深度为一层
