# 审查

对 `HEAD` 和用户提供的固定点之间的差异进行双轴审查：

- **标准** — 代码是否符合此仓库记录的编码标准？
- **规格** — 代码是否忠实地实现了原始 Issue / PRD / 规格？

两个轴作为**并行子智能体**运行，这样它们不会污染彼此的上下文，然后此技能汇总它们的发现。

问题跟踪器应该已经提供给你——如果 `docs/agents/issue-tracker.md` 缺失，请运行 `/setup-matt-pocock-skills`。

## 流程

### 1. 确定固定点

用户说的任何东西都是固定点——提交 SHA、分支名、标签、`main`、`HEAD~5` 等。不要有意见；直接采用。如果他们没有指定，询问："对照什么进行审查——一个分支、一个提交或 `main`？"在有了之前不要继续。

捕获一次 diff 命令：`git diff <fixed-point>...HEAD`（三点，这样比较是对合并基础进行的）。也通过 `git log <fixed-point>..HEAD --oneline` 记录提交列表。

### 2. 识别规格来源

按此顺序寻找原始规格：

1. 提交消息中的 Issue 引用（`#123`、`Closes #45`、GitLab `!67` 等）— 通过 `docs/agents/issue-tracker.md` 中的工作流获取。
2. 用户作为参数传递的路径。
3. `docs/`、`specs/` 或 `.scratch/` 下与分支名或功能匹配的 PRD/规格文件。
4. 如果找不到，询问用户规格在哪里。如果他们说没有，**Spec** 子智能体将跳过并报告"无规格可用"。

### 3. 识别标准来源

仓库中任何记录代码应该如何编写的文件。常见位置：

- `CLAUDE.md`、`AGENTS.md`
- `CONTRIBUTING.md`
- `CONTEXT.md`、`CONTEXT-MAP.md`、每个上下文的 `CONTEXT.md` 文件
- `docs/adr/`（架构决策就是标准）
- `.editorconfig`、`eslint.config.*`、`biome.json`、`prettier.config.*`、`tsconfig.json`（机器强制标准——记录但不重复检查工具已检查的内容）
- 仓库根目录或 `docs/` 下的任何 `STYLE.md`、`STANDARDS.md`、`STYLEGUIDE.md` 或类似文件

收集文件列表。**Standards** 子智能体将读取它们。

### 4. 并行启动两个子智能体

发送一条消息，包含两个 `Agent` 工具调用。使用 `general-purpose` 子智能体用于两者。

**Standards 子智能体提示** — 包含：

- 完整的 diff 命令和提交列表。
- 你在步骤 3 中找到的标准源文件列表。
- 简报："阅读标准文档。然后阅读差异。报告——在每个相关文件/块处——差异违反记录标准的每个地方。引用标准（文件 + 规则）。区分硬性违规和判断调用。跳过工具已强制执行的内容。400 词以内。"

**Spec 子智能体提示** — 包含：

- 完整的 diff 命令和提交列表。
- 规格的路径或获取的内容。
- 简报："阅读规格。然后阅读差异。报告：(a) 规格要求的缺失或不完整的需求；(b) 差异中规格未要求的行为（范围蔓延）；(c) 看起来已实现但实现看起来错误的需求。为每个发现引用规格行。400 词以内。"

如果规格缺失，跳过 Spec 子智能体并在最终报告中注明。

### 5. 汇总

在 `## 标准` 和 `## 规格` 标题下展示两个报告，逐字或轻度清理。不要合并或重新排列发现——两个轴特意分离，以便用户可以独立查看它们。

以一行摘要结尾：每个轴的发现总数，以及标记的最严重单项问题（如果有）。

## 为什么是两个轴

一个变更可以通过一个轴但未通过另一个：

- 遵循每个标准但实现了错误事物的代码 → **标准通过，规格失败。**
- 精确地完成了 Issue 要求但违反项目约定的代码 → **规格通过，标准失败。**

分开报告它们阻止一个轴掩盖另一个轴。
